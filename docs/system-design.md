## 1. نظرة عامة على النظام

### 1.1 الغرض من المنصة

المنصة هي تطبيق ويب متكامل الجانبين (Full-Stack) مبني على إطار عمل Next.js 16، يهدف إلى تمكين المستخدمين من تصوّر المنتجات في بيئاتهم الحقيقية باستخدام الذكاء الاصطناعي، فضلاً عن تقديم تجارب عرض تفاعلية ثلاثية الأبعاد. تتكوّن المنصة من أربعة أنظمة رئيسية متكاملة:

| النظام | الوصف |
|--------|-------|
| **معاينة الغرفة (Room Preview)** | يتيح للمستخدم رفع صورة غرفته والحصول على معاينة فورية بالذكاء الاصطناعي تُظهر المنتج داخل الغرفة |
| **العارض البانورامي (Panorama Viewer)** | جولات 360 درجة تفاعلية مع نقاط حرارية (Hotspots) للتنقل بين المشاهد |
| **صالة العرض (Showroom)** | عرض دوّار ثلاثي الأبعاد (Carousel) للمنتجات مُحسَّن لشاشات العرض الكبيرة |
| **لوحة الإدارة (Admin Dashboard)** | نظام تحليلات وإدارة جلسات للمشغّلين |

### 1.2 المكدس التقني (Technology Stack)

| الطبقة | التقنيات |
|--------|---------|
| إطار العمل | Next.js 16 (App Router)، React 19 |
| اللغة | TypeScript 5 |
| قاعدة البيانات | PostgreSQL + Prisma ORM |
| التخزين المؤقت / النشر-الاشتراك | Redis (اختياري، مكتبة ioredis) |
| الرسومات ثلاثية الأبعاد | Three.js، @react-three/fiber، GSAP، Photo Sphere Viewer |
| الذكاء الاصطناعي | Google Gemini API |
| تخزين الملفات | Cloudflare R2 / AWS S3 (الإنتاج)، نظام ملفات محلي (التطوير) |
| المصادقة | رموز JWT/HMAC مخصصة (Web Crypto API) |
| التسجيل | Pino |
| المراقبة | Sentry |
| التنسيق | Tailwind CSS 4 |
| الاختبار | Vitest، Playwright |
| النشر | Vercel (Pro للمهلة الزمنية 300 ثانية) |

### 2.1 نمط البنية

يعتمد النظام على نمط **البنية المعمارية للخدمات الدقيقة الأحادية (Modular Monolith)**، حيث يتم تجميع كل الوظائف في تطبيق Next.js واحد مع فصل واضح للمسؤوليات بين الوحدات. هذا النهج يوفر بساطة التشغيل مع الحفاظ على قابلية الصيانة.

**المبادئ المعمارية الأساسية:**
- **الفصل بين الاهتمامات (Separation of Concerns):** كل وحدة في `lib/` تملك نطاقاً محدداً
- **الفشل الرشيق (Graceful Degradation):** Redis اختياري، والنظام يعمل بدونه
- **الأمان الافتراضي (Secure by Default):** المصادقة مطلوبة على جميع نقاط النهاية الحساسة
- **قابلية الملاحظة (Observability):** Sentry + Pino + تحليلات الأحداث

### 4.2 تدفق الجولة البانورامية

```
المستخدم → يختار الجولة → يُحمَّل PanoramaViewer
         → يتفاعل مع نقاط الحرارة (Hotspots)
         → ينتقل بين المشاهد
         → تُحدَّث عناوين URL ديناميكياً
         → [اختياري] وضع المحرر لإضافة نقاط الحرارة
```

### 4.3 آلية عمل الجلسة والبوابة (Gate)

قبل الوصول إلى واجهة الجهاز المحمول، يمر المستخدم بـ**بوابة جمع البيانات** (`/room-preview/gate/{sessionId}`) التي تجمع:
- **للعملاء:** الاسم، رقم الهاتف، الدور
- **للموظفين:** الاسم، رمز الموظف

تُحفظ هذه البيانات في نموذج `UserSession` وتُستخدم في التحليلات وإعداد التقارير.

### 5.2 بنية آلة الحالة (State Machine)

تتتبع كل جلسة دورة حياة صارمة من خلال آلة حالة محددة في `lib/room-preview/session-machine.ts`:

```
waiting_for_mobile
       │
       ▼ (POST /connect)
mobile_connected
       │
       ▼ (POST /room)
room_selected
       │
       ▼ (POST /product)
product_selected
       │
       ▼ (POST /render - التحقق من الحد)
ready_to_render ──► rendering ──┬──► result_ready
                                │
                                └──► failed

(أي حالة) ──► expired (عند انتهاء المهلة)
```

**الخصائص الأساسية للآلة:**
- لا يمكن قلب الحالة للخلف (الانتقالات أحادية الاتجاه)
- تُحفظ كل انتقال حالة في قاعدة البيانات بشكل فوري
- الحالات "المقفلة" (`rendering`, `result_ready`, `failed`, `expired`) تمنع التعديلات الإضافية

### 5.3 حارس نقاط النهاية (API Guard)

الملف: `lib/room-preview/api-guard.ts`

يوفر طبقة أمان موحدة لجميع نقاط نهاية `room-preview`:
1. **التحقق من الرمز المميز:** التحقق من HMAC-SHA256 لرموز الجلسة
2. **التحقق من وجود الجلسة:** التحقق من قاعدة البيانات
3. **التحقق من صلاحية الجلسة:** رفض الجلسات المنتهية الصلاحية
4. **التحقق من الحالة:** التحقق من أن الحالة الحالية تسمح بالعملية المطلوبة

## 7. الاتصال في الوقت الفعلي

### 7.1 بنية Server-Sent Events (SSE)

يعتمد النظام على **Server-Sent Events (SSE)** بدلاً من WebSockets لأسباب وجيهة:
- **أحادية الاتجاه:** التحديثات تنتقل من الخادم إلى العميل فقط (مناسب لحالة الاستخدام)
- **التوافق مع Vercel:** لا يتطلب بنية تحتية إضافية
- **إعادة الاتصال التلقائية:** المتصفح يعيد الاتصال تلقائياً عند الانقطاع

```
                    ┌─────────────────────────────────────┐
                    │          طبقة Pub/Sub               │
                    │                                     │
                    │  ┌───────────────┐                  │
                    │  │ Redis Pub/Sub │ ← إذا كان        │
                    │  │ (الإنتاج)     │   REDIS_URL      │
                    │  └───────┬───────┘   متاحاً         │
                    │          │                          │
                    │  ┌───────▼───────┐                  │
                    │  │ In-Memory Bus │ ← احتياطي        │
                    │  │ (خادم واحد)   │   (التطوير)     │
                    │  └───────────────┘                  │
                    └────────────┬────────────────────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
    ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
    │  SSE Stream  │   │  SSE Stream  │   │  SSE Stream  │
    │  (Screen 1)  │   │  (Screen 2)  │   │  (Mobile)    │
    └──────────────┘   └──────────────┘   └──────────────┘
```

### 7.2 مواصفات نقطة نهاية SSE

**المسار:** `GET /api/room-preview/sessions/{sessionId}/events`

**رأس الاستجابة:**
```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
X-Accel-Buffering: no
```

**تنسيق الحدث:**
```
event: session_updated
data: {"id":"cls...","status":"rendering","mobileConnected":true,...}

: heartbeat (كل 5 ثوانٍ)
```

**آلية إعادة الاتصال:**
- `retry: 3000` (3 ثوانٍ قبل إعادة المحاولة)
- العميل يُعيد الاتصال تلقائياً عند انقطاع الشبكة
- الخادم يُرسل حالة الجلسة الحالية فور الاتصال

### 7.3 آلية القفل الموزع (Distributed Lock)

لمنع التصيير المتزامن على نفس الجلسة:

```
طلب التصيير
     │
     ▼
tryAcquireLock(sessionId, TTL=300s)
     │
  ┌──┴──┐
  │Redis│ ← SET NX EX
  └──┬──┘
     │
  ┌──┴──────────────┐
  │ القفل محجوز؟   │
  └──┬──────────────┘
     │
  نعم ──► HTTP 429 (قيد معدل الاستخدام)
     │
  لا ──► المضي في التصيير
           │
           ▼
    [عملية Gemini]
           │
           ▼
    releaseLock(sessionId)
```

## 9. قابلية التوسع

### 9.1 آليات التوسع الأفقي

**طبقة التطبيق (Next.js / Vercel):**
- يستقبل كل طلب دالة serverless مستقلة
- يمكن تشغيل آلاف الدوال بالتوازي
- لا توجد حالة محلية محفوظة في ذاكرة الخادم (باستثناء الاحتياطي في التطوير)

**قاعدة البيانات (PostgreSQL):**
- العمليات الذرية (`tryIncrementRenderCount`, `tryClaimRenderingSlot`) تمنع حالات السباق
- الفهارس على أعمدة الاستعلام المتكررة (`status`, `expiresAt`, `userSessionId`, `createdAt`)
- يمكن استخدام اتصال مجمّع (Connection Pooling) عبر PgBouncer أو Prisma Accelerate

**Redis:**
- يمنع مشكلة الحالة المشتركة في بيئات متعددة الخوادم
- يوفر آلية Pub/Sub مقاومة للتوزيع
- استخدام Redis Cluster للتوسع الأفقي في الإنتاج

**تخزين الملفات (R2/S3):**
- تخزين كائنات (Object Storage) قابل للتوسع تلقائياً
- لا يوجد حد على حجم التخزين
- شبكة CDN مدمجة في Cloudflare R2

### 9.2 آليات تحديد المعدل (Rate Limiting)

| المستوى | القيد | الهدف |
|---------|-------|-------|
| إنشاء الجلسة | 10 جلسات/دقيقة لكل IP | منع الإساءة الجماعية |
| تشغيل التصيير | جلستان/جلسة كحد أقصى | التحكم في تكاليف API |
| فترة التهدئة | 5 دقائق بين التصييرات | منع الضغط على نموذج AI |
| القفل الموزع | قفل واحد لكل جلسة | منع التصيير المتزامن |

### 9.3 التوسع المستقبلي المُخطط

- **نظام قوائم الانتظار (Job Queue):** BullMQ أو Celery لمعالجة عمليات التصيير بشكل غير متزامن
- **تخزين مؤقت (Caching):** Redis لنتائج التصيير المتطابقة (نفس الغرفة + نفس المنتج)
- **تعدد المناطق (Multi-Region):** نشر في مناطق جغرافية متعددة على Vercel

## 10. الأمان

### 10.1 نموذج المصادقة

**رموز الجلسة (Session Tokens):**
- **الخوارزمية:** HMAC-SHA256 (Web Crypto API)
- **البنية:** `base64url(header).base64url(payload).signature`
- **النقل:** رأس HTTP `x-session-token` أو ملف تعريف ارتباط `rp-screen-token` (HttpOnly)
- **الصلاحية:** مرتبطة بدورة حياة الجلسة

**مصادقة الإدارة:**
- رمز JWT مخصص مع توقيع HMAC-SHA256
- صلاحية 8 ساعات
- يُخزَّن في ملف تعريف ارتباط HttpOnly محمي
- بيانات الاعتماد من متغيرات البيئة فقط

### 10.2 رؤوس الأمان (Security Headers)

| الرأس | القيمة | الهدف |
|-------|--------|-------|
| `Strict-Transport-Security` | `max-age=63072000; includeSubDomains` | إجبار HTTPS |
| `X-Frame-Options` | `DENY` | منع Clickjacking |
| `X-Content-Type-Options` | `nosniff` | منع MIME Sniffing |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | خصوصية الإحالة |
| `Permissions-Policy` | `camera=(), microphone=()` | رفض الوصول للأجهزة |

### 10.3 التحقق من صحة المدخلات

- **أنواع MIME للصور:** قائمة بيضاء صارمة (`image/jpeg`, `image/png`, `image/webp`)
- **حجم الصورة:** حد أقصى 10 ميجابايت مع ضغط من جانب العميل
- **معرّفات المنتجات:** التحقق من وجود المعرّف في القائمة الثابتة
- **معرّفات الجلسات:** تنسيق CUID محدد
- **بيانات البوابة:** مطلوبة واسم غير فارغ

### 10.4 حماية نقاط النهاية

```
طلب API
    │
    ▼
[رأس Authorization / ملف تعريف الارتباط]
    │
    ▼
api-guard.ts
    ├── التحقق من الرمز المميز
    ├── التحقق من وجود الجلسة في DB
    ├── التحقق من صلاحية الجلسة
    └── التحقق من انتقالات الحالة المسموح بها
         │
         ▼
    [معالج نقطة النهاية]
```

### 10.5 حماية المفاتيح والأسرار

- جميع المفاتيح السرية في متغيرات البيئة (لا تُخزَّن في الكود)
- مفتاح تنظيف الجلسات (`CLEANUP_SECRET`) يحمي نقطة نهاية الكرون
- مفاتيح Sentry تُكشف فقط من جانب العميل (`NEXT_PUBLIC_`)
- ملفات `.env` مستبعدة من Git عبر `.gitignore`

## 11. القيود الحالية

### 11.1 قيود بنيوية

| القيد | الوصف | التأثير |
|-------|-------|---------|
| **SSE أحادية الاتجاه** | لا يمكن للعميل إرسال رسائل عبر قناة SSE | لا توجد ثنائية الاتجاه في الوقت الفعلي |
| **In-Memory Pub/Sub** | احتياطي SSE يعمل فقط مع خادم واحد | يتطلب Redis في بيئات متعددة الخوادم |
| **تخزين محلي في التطوير** | `public/uploads/` غير قابل للمشاركة | لا يتوسع في بيئات الإنتاج |
| **جلستا تصيير فقط** | كل جلسة محدودة بتصييرين | قد يُحبط المستخدم الراغب في تجربة منتجات متعددة |
| **مهلة Vercel 300 ثانية** | يتطلب Vercel Pro | التصييرات البطيئة جداً قد تفشل |

### 11.2 قيود نموذج الذكاء الاصطناعي

| القيد | الوصف |
|-------|-------|
| **وقت الاستجابة** | 10-30 ثانية في المتوسط لكل تصيير |
| **جودة التصيير** | تتفاوت الجودة بناءً على إضاءة الغرفة وزاوية التصوير |
| **نسبة الأبعاد** | قد تختلف نسبة أبعاد صورة الإخراج قليلاً (< 2%) |
| **اعتماد الموفر الوحيد** | الاعتماد على Google Gemini فقط حالياً |
| **عدم التحكم في التنسيق** | لا يمكن تحديد موضع المنتج بدقة |

### 11.3 قيود التشغيل

- **تنظيف الجلسات:** يعتمد على cron job خارجي لتنظيف الجلسات المنتهية
- **الإشعارات:** لا يوجد نظام إشعارات للتصييرات الفاشلة
- **المراقبة:** المراقبة محدودة بـ Sentry (لا يوجد لوحة مقاييس مخصصة)
- **النسخ الاحتياطي:** لا توجد استراتيجية نسخ احتياطي موثقة لقاعدة البيانات
